## 二、详细指标说明

### 2.1 指标定义与计算公式


| 指标                  | 含义                                     | 计算公式                                             |
| ------------------- | -------------------------------------- | ------------------------------------------------ |
| **LayerAcc**        | 检索到的文档是否命中正确架构层（L1-L5/CC）              | `正确命中层数 / 总查询数`                                  |
| **Recall@K**        | Top-K 文档中覆盖 Ground Truth key_files 的比例 | `∩(retrieved[:K], GT_files) /                    |
| **MRR**             | 第一个正确文档的排名倒数平均                         | `mean(1/rank_i)` if found, else 0                |
| **Path Validity**   | 返回文档路径在磁盘上实际存在的比例                      | `存在路径数 / 返回文档总数`                                 |
| **Memory Recall**   | LLM 从上下文能召回知识点的比例（模拟评分）                | `min(retrieved_keys ∩ GT_keys /                  |
| **Info Density**    | 单位 Token 承载的信息量                        | `Recall@5 × GT_coverage / context_tokens × 1000` |
| **Context Tokens**  | 系统返回上下文的 Token 数（按字符/4 估算）             | `len(context_text) / 4`                          |
| **Token/R@5**       | 每单位 Recall@5 消耗的 Token 数（越低越好）         | `mean(context_tokens) / mean(recall@5)`          |
| **Actionability**   | 返回内容对 Agent 的可操作性（0=不可用，3=直接可用）        | 规则打分：含代码 +1，含路径 +1，含示例 +1                        |
| **Latency P50/P95** | 检索延迟中位数和 95 分位数                        | 按毫秒记录，取分位数                                       |


### 2.2 统计检验

- **McNemar 检验（层级准确率）**：Ontology vs Markdown：p≈0.024；Ontology vs HybridRAG：p≈0.024
  - 解读：在 45 条查询上，Ontology 胜出 Markdown/RAG 各 18 次，对方仅胜出 2 次，差异在 95% 置信水平下显著
- **Recall@5 差值分布**（Ontology - Markdown）：均值 +0.296，说明平均每条查询 Ontology 多覆盖约 30% 的 Ground Truth 文件

### 4.2 分类别优势分析

**Ontology MMS 明显领先（优势 >20pp）的场景**：

- D 类（对抗/混淆）：LayerAcc +80pp，R@5 +63.3pp — 本体规则对语义混淆有强鉴别力
- C 类（代码实现）：R@5 +50pp — 精确映射到实现文件
- E 类（混合 GT）：R@5 +33.4pp — 跨层文档的联合检索
- B 类（调试问题）：LayerAcc +42.8pp — 调试意图被正确路由到 L2/L4 层

**Markdown/Hybrid RAG 具有相对优势的场景**：

- C/D 类 **Memory Recall**：Markdown 在返回整个模块文件时包含更多背景信息（但文件级召回会带来大量噪声）
- F 类 **R@5**：Markdown 和 RAG 各得 20%，Ontology 仅 10%——纯概念性文档查询中，BM25 关键词匹配对纯文本类文档有优势
- **Actionability**：Markdown/RAG 得分略高（2.4-2.6 vs 2.1），原因是它们倾向返回含代码的实现文件

## 六、公平性说明


| 维度                     | 处理方式                                                                                           |
| ---------------------- | ---------------------------------------------------------------------------------------------- |
| **语料库统一**              | 三系统使用相同 `docs/` 语料（MD Index 和 RAG 索引相同文档集）                                                     |
| **Ground Truth 独立**    | GT `key_files` 由人工标注，不依赖任何系统的输出                                                                |
| **任务分布均衡**             | V1（MDP 专有 25条） + V2（通用 SE 20条），8 类覆盖架构/调试/代码/对抗/混合/概念/通用/多跳                                    |
| **评估指标中性**             | Recall@K、MRR、LayerAcc 均不利于特定系统设计                                                               |
| **F 类偏差补偿**            | 纯概念查询 Ontology 占优（LayerAcc 80% vs 20%）但 R@5 落后，说明层级路由正确但文件未全覆盖——这是 **当前 Ontology 的已知局限**，已如实呈现 |
| **Markdown MemRec 偏高** | 因返回整文件，Memory Recall 覆盖度高，但同时带入大量噪声（InfoDensity=0.06），不代表精度优势                                  |
| **统计显著性**              | McNemar p=0.024 < 0.05，Ontology 优势在统计上显著                                                       |

## 八、问题与局限

### 8.1 已识别的 Ontology 弱点

1. **纯文档概念查询（F 类）**：R@5 仅 10%，低于 Markdown（20%）。原因：`docs/architecture/` 下的概念性 Markdown 文件未完全纳入 slot 注入规则。
  - **建议修复**：为架构概念类操作（`rule_arch_concept_review`）增加 `frontend_page_map.md`、`e2e_traceability.md` 等文件的 hint_files。
2. **Actionability 略低（2.13 vs 2.47-2.53）**：Ontology 倾向返回规约/记忆文档而非代码文件，导致"含代码"得分较低。
  - **建议修复**：对 `operation=implement` 类查询，优先注入对应的实现文件路径。
3. **通用 SE 任务（G/H 类）表现尚可但不突出**：G 类 R@5=20%，H 类 R@5=26.7%，仍有提升空间。
  - **建议**：扩充通用 SE 任务规则覆盖，或引入 Agent 记忆学习机制。

### 8.2 Hybrid RAG 局限

- P50 延迟 383ms（对于实时 AI 编码场景不友好）
- 在 MDP 专有任务上 R@5=0%（向量模型未能有效区分专有架构概念）
- 依赖 ES + Milvus 基础设施，本地开发环境成本高

### 8.3 Markdown BM25 局限

- 所有维度与 Hybrid RAG 相近，但延迟更低（22ms）
- 无层级理解能力，LayerAcc 仅 24.4%
- V1 数据集 R@5=0%（专有领域关键词未覆盖）
