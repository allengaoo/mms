# BIZ-002 · 数据同步管道的创建与调度执行流程

## 业务场景（Business Context）

数据工程师需要将外部数据库（MySQL、PostgreSQL、S3 等）的数据定期同步到 MDP 平台，
映射为本体对象（ObjectInstance），并触发向量化索引。

**用户**：数据工程师（Data Engineer）
**问题**：如何配置一个可靠的、可监控的数据同步管道，从外部源到 MDP 本体层？

## 核心流程（Core Flow）

**配置连接器（Connector）：**

1. 数据工程师在"数据连接"模块填写连接器配置（类型/主机/凭据）
2. 调用 `POST /api/v1/connectors` 创建连接器
3. `ConnectorService.create()` 存储加密凭据（`EncryptionService`）
4. 调用 `GET /api/v1/connectors/{id}/test` 测试连通性
5. 连通性测试通过后，连接器 `status=active`

**创建同步任务（SyncJob）：**

1. 选择连接器、目标对象类型、字段映射规则
2. 调用 `POST /api/v1/sync-jobs`
3. 配置调度策略（Cron 表达式 / 实时触发 / 手动触发）
4. `SyncJobService.create()` 写入 `sys_sync_jobs`
5. `SchedulerService` 注册 Cron 任务

**执行同步（Ingestion）：**

1. Scheduler（Leader Election 保护）到时触发 `IngestionWorker`
2. `IngestionWorker` 使用 `JobExecutionScope` 管理生命周期
3. 流式读取数据（`yield`），避免 OOM
4. 每条记录通过 `normalize_record()` 归一化
5. 发送到 Kafka topic `data.ingestion.raw`
6. `IndexingWorker` 消费 Kafka，转换为 ObjectInstance 写入 MySQL
7. 异步触发向量化（`VectorWorker` → Milvus）

## 参与者（Actors & Roles）

- **数据工程师**：配置连接器和同步任务
- **Scheduler**：定时触发（Leader Election 保护，只有 Leader 执行）
- **IngestionWorker**：从外部源流式读取数据
- **Kafka**：解耦读取与处理
- **IndexingWorker**：将 Raw 数据转为 ObjectInstance
- **VectorWorker**：异步生成向量嵌入

## 业务规则（Constraints & Rules）

- 连接器凭据必须加密存储（`EncryptionService`，标记为 `sensitive`）
- 同步任务执行必须用 `JobExecutionScope`，禁止散落 try/except 管理状态
- 流式读取：禁止 `fetchall()` / `read()`，必须用 `yield` 生成器
- Kafka 消息必须先调 `normalize_record(row)` 再发送（防止原生类型序列化失败）
- 同一 SyncJob 同一时间只能有一个实例在运行（幂等锁：Redis SET NX EX）
- 同步频率不能超过 `SystemConfig.get("max_sync_frequency_minutes")`

## 数据流（Data Flow）

```
Scheduler → IngestionWorker.run(job_id)
              → JobExecutionScope.__enter__()（set status=running）
              → Connector.fetch_stream() → yield rows
                  → normalize_record(row)
                  → Kafka.send("data.ingestion.raw", avro_encoded)
              → JobExecutionScope.__exit__()（set status=success/failed）

Kafka → IndexingWorker.on_message(raw_record)
          → map_to_object_instance(mapping_rules)
          → [MySQL] INSERT sys_object_instances
          → [Kafka] emit("data.indexed", instance_id)

Kafka → VectorWorker.on_indexed(instance_id)
          → generate_embedding(text_fields)
          → [Milvus] upsert_vector(instance_id, embedding)
```

## 边界情况（Edge Cases）

- **连接超时**：外部数据库连接超时（Tenacity 重试 3 次，超过则标记 job_status=failed）
- **Schema 变更**：源表新增列 → 通过 `DataCatalog` 自动检测，发送 `schema_drift` 告警
- **重复执行**：Scheduler 重启后重复触发 → Redis 幂等锁防止重复
- **大批量数据**：单次同步超过 100 万条 → 自动分批，每批 5000 条
- **Leader 宕机**：正在执行的 Worker 进程死亡 → Redis lock 超时后新 Leader 接管
- **Avro Schema 不匹配**：消息格式变化 → Schema Registry 版本控制，向前兼容
