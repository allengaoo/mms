# BIZ-001 · 全域对象类型定义的创建与修改流程

## 业务场景（Business Context）

全域资产库（Global Registry）中的对象类型（ObjectTypeDef）是 MDP 平台的核心元数据资产。
对它们的创建/修改是高风险操作，因为任何变更都可能影响所有引用该对象类型的场景（Scenario）、
应用（App）和数据管道（Pipeline）。

**用户**：数据架构师（Architect）
**问题**：如何安全地创建和修改全局对象类型，同时最小化对下游的影响？

## 核心流程（Core Flow）

**创建新对象类型：**

1. 架构师在"全域资产库"UI 点击"新建对象类型"
2. 填写名称（英文 key）、显示名称、描述、属性列表
3. 系统调用 `POST /api/v1/object-types`
4. `ObjectTypeService.create()` 验证：名称唯一性（租户内）、属性类型合法性
5. 写入 `meta_object_defs` 表（含 `tenant_id`、`status=draft`）
6. 触发 Kafka 事件 `ontology.object_type.created`（供搜索索引、Scene 感知）
7. 返回 201 Created + 新对象类型 ID

**修改已发布对象类型：**

1. 架构师点击"编辑"
2. UI 显示"影响分析"弹窗（调用 `GET /api/v1/object-types/{id}/impact`）
3. 影响分析结果展示：受影响的 Scenario 数量、Pipeline 数量
4. 架构师确认后提交变更请求（Change Request, CR）
5. CR 经过审批流（Approval Flow）后才能合并
6. 合并时触发增量重建（Incremental Reindex）

## 参与者（Actors & Roles）

- **架构师（Architect）**：有 `ont:object:edit` 权限的用户，负责全局资产定义
- **业务用户（Business User）**：只读，通过 Scenario 使用对象类型
- **审批者（Approver）**：拥有 `gov:cr:approve` 权限，审批修改请求
- **Kafka**：解耦写操作与搜索索引重建
- **Elasticsearch / Milvus**：接收事件后异步重建索引

## 业务规则（Constraints & Rules）

- 对象类型名称在同一 `tenant_id` 下全局唯一（数据库唯一约束）
- 只有 `status=published` 的对象类型才能被 Scenario 引用
- 修改 `published` 状态的对象类型必须走 CR 流程（不可直接 update）
- 删除操作必须是软删除（`status=archived`），且前置检查无活跃引用
- 新增属性是向前兼容的，不需要 CR；删除/重命名属性需要 CR

## 数据流（Data Flow）

```
架构师 → [API] POST /object-types
         → [Service] ObjectTypeService.create(ctx, payload)
             → validate(name_unique, attr_types)
             → [DB] INSERT meta_object_defs (tenant_id, status=draft)
             → [Kafka] emit("ontology.object_type.created", payload)
                 → [Worker] SearchIndexWorker.on_created()
                     → [ES/Milvus] upsert_index()
         → return 201 {code:201, data:{id, key, status}, meta:{}}
```

## 边界情况（Edge Cases）

- **名称冲突**：同一租户下已有同名对象类型 → 返回 `E_ONT_DUPLICATE_KEY`（422）
- **循环引用**：新建对象类型的属性引用自身 → 返回 `E_ONT_CIRCULAR_REF`（422）
- **并发修改**：两个架构师同时修改同一对象类型 → 乐观锁（version 字段）冲突 → 409
- **删除有引用的对象**：`DELETE /object-types/{id}` 时仍有 Scenario 引用 → 返回 `E_ONT_IN_USE`（409）
- **跨租户隔离**：查询 / 修改必须通过 `ctx.tenant_id` 过滤，防止跨租户访问
