# BIZ-003 · 多租户隔离与配额治理规则

## 业务场景（Business Context）

MDP 是多租户 SaaS 平台。每个租户（Tenant）拥有独立的命名空间、数据、权限和资源配额。
租户间必须完全隔离，任何代码逻辑泄露都可能导致严重的数据安全事故。

**用户**：平台管理员（Platform Admin）、企业 IT 管理员（Tenant Admin）
**问题**：如何在技术层和业务层双重保障多租户隔离？

## 核心流程（Core Flow）

**租户创建：**

1. 平台管理员调用 `POST /api/v1/admin/tenants`
2. 创建租户记录（`tenant_id` 生成 UUID v4）
3. 初始化默认配额（`TenantQuota`：对象数量上限、存储上限、API 调用次数）
4. 创建默认管理员用户（发送邀请邮件）
5. 初始化 RBAC 角色（Admin / Editor / Viewer）

**资源访问隔离（每次 API 调用）：**

1. JWT Token 携带 `tenant_id` 和 `user_id`
2. `SecurityMiddleware` 解析 Token，构建 `SecurityContext`（含 tenant_id, roles, permissions）
3. 所有 Service 方法接收 `ctx: SecurityContext` 作为首参
4. 所有 DB 查询必须包含 `WHERE tenant_id = ctx.tenant_id`（RLS）
5. Response 数据通过 `ResponsePruner` 过滤敏感字段

**配额检查：**

1. 写操作前，`QuotaEnforcer.check(ctx, resource_type, delta)` 验证配额
2. 当前用量通过 Redis 缓存（`quota:{tenant_id}:{resource}` Hash）
3. 超出配额 → 返回 `E_QUOTA_EXCEEDED`（429）
4. 每次写成功后，Redis 计数器 +1

## 参与者（Actors & Roles）

- **平台管理员**：跨租户管理（有特殊 `platform:*:admin` 权限）
- **租户管理员**：在自己租户内管理用户、角色、配额
- **SecurityMiddleware**：在入口构建并注入 `SecurityContext`
- **QuotaEnforcer**：在每次写操作前检查配额
- **AuditService**：记录所有写操作（含失败的）

## 业务规则（Constraints & Rules）

**隔离规则（违反会导致数据泄露）：**

- `tenant_id` 必须从 `SecurityContext` 获取，禁止从请求 body/query 传入
- 所有 SELECT 语句必须包含 `WHERE tenant_id = :tenant_id`（使用 Repository 模式）
- 跨租户查询只允许平台管理员角色（`platform:data:admin`）

**配额规则：**

- 对象实例数量上限：`SystemConfig.get("default_quota_objects")`（默认 10 万）
- 存储空间上限：`SystemConfig.get("default_quota_storage_gb")`（默认 100GB）
- API 调用次数：按天滚动窗口，Redis INCR + EXPIRE

**审计规则：**

- 每次 WRITE 操作（create/update/delete）必须调 `AuditService.log(ctx, action, resource_id)`
- 审计日志不可删除（append-only），写入专用 MySQL 表 `sys_audit_logs`

**敏感字段规则：**

- `sensitive=true` 的字段（如密码、Token）必须用 `EncryptionService` 加密存储
- Response 中必须用 `response_model_exclude` 排除 sensitive 字段

## 数据流（Data Flow）

```
HTTP Request → SecurityMiddleware
               → parse JWT (tenant_id, user_id, roles)
               → build SecurityContext
               → inject into request.state

Controller → Service.create(ctx: SecurityContext, payload)
               → QuotaEnforcer.check(ctx, "objects", delta=1)
                   → Redis GET quota:{tenant_id}:objects
                   → if current + delta > limit → raise E_QUOTA_EXCEEDED
               → Repository.create(ctx, entity)
                   → INSERT ... WHERE tenant_id = ctx.tenant_id
               → AuditService.log(ctx, "create", entity.id)
               → QuotaEnforcer.increment(ctx, "objects")
                   → Redis INCR quota:{tenant_id}:objects
```

## 边界情况（Edge Cases）

- **Token 中 tenant_id 伪造**：JWT 签名验证 + 数据库二次确认（关键操作）
- **Redis 宕机**：配额检查降级为数据库查询（性能损耗可接受）
- **配额 Redis 与实际数量不同步**：定期 `QuotaReconciler` 任务重计数（每小时）
- **平台管理员误操作**：平台管理员操作也生成 audit log，且需要双人审批
- **租户删除**：软删除（`status=suspended`）→ 数据保留 30 天 → 彻底删除
