# AD-006 · JobExecutionScope 作为 Worker 唯一生命周期管理机制

## 决策
所有 Worker 必须通过 `JobExecutionScope` 异步上下文管理器管理 Job 状态（pending→running→completed/failed）。

## 背景
早期 Worker 代码中散落大量 try/except 管理 `job.status`，导致异常路径状态不一致（Job 卡在 running 状态）。

## 替代方案对比
| 方案 | 优劣 |
|:---|:---|
| A. 各 Worker 自行管理状态 | 已证明不可维护，状态不一致 |
| **B（选择）** | JobExecutionScope 统一进入/退出，含自动状态翻转和错误捕获 |

## 影响
```python
# 所有 Worker 结构必须是：
async with JobExecutionScope(job_id, ctx) as scope:
    # Worker 业务逻辑
    scope.set_metric("rows_processed", count)
# 退出时自动翻转 completed/failed 状态
```
- ❌ 禁止在 Worker 中散落 try/except 来管理 Job 状态
